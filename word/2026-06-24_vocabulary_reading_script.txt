abstraction.

The harness exposed stable abstractions instead of raw internal details.

internal structure.
The organization inside a system that supports reliable progress.
Agents need internal structure when tasks span many steps.
Weak internal structure can make an otherwise capable model fail.

underspecified environment.
An environment that lacks the tools, context, or structure needed for reliable work.
The agent failed because the environment was underspecified.
An underspecified environment can be mistaken for weak model capability.

A point where progress depends too heavily on human inspection or action.
Manual quality assurance can become a human bottleneck.

inspect and act on.

repository knowledge.

system of record.
The authoritative place where important information is stored and maintained.

structured documentation.
Documentation organized in a predictable format that is easy to search, maintain, and validate.

versioned documentation.
Documentation tracked with versions, usually alongside code changes.

static benchmark.
A fixed test that does not simulate interactive deployment conditions.
Static benchmarks may miss failures that appear in realistic workflows.
A model can score well on static benchmarks and still fail in deployment simulation.

real deployment conditions.
The conditions a system faces when used in production by real users.
A useful simulation should mirror real deployment conditions.

higher-fidelity simulation.
A simulation that more closely matches real usage, tasks, constraints, and feedback loops.
Higher-fidelity simulations can surface risks that simple tests miss.
The team used a higher-fidelity simulation to evaluate long-running workflows.

surface risks.
To reveal risks that were previously hidden or hard to observe.

feedback handling.
The process of receiving, interpreting, and responding to feedback.
